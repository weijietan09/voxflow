# 设计笔记

一些实现取舍与背后的想法，主要留给自己和后来者。

## 为什么用流匹配

扩散模型在语音上效果好，但推理步数多。条件流匹配（CFM）学习一个把噪声直线
搬运到数据的向量场，配合最优传输路径，推理时用很少的欧拉步就能得到可用结果，
这对交互式声音克隆很关键。仓库同时保留了 DDPM 实现，主要为了：

1. 验证同一个向量场估计器接口对"预测噪声"和"预测速度场"两种目标都够通用；
2. 方便在同条件下做对照实验。

训练目标（OT-CFM）：

```
t ~ U(0, 1),  x0 ~ N(0, I)
x_t = (1 - (1 - σ_min) t) · x0 + t · x1
u_t = x1 - (1 - σ_min) · x0
loss = || v_θ(x_t, t, cond) - u_t ||²
```

## 为什么说话人编码器用 LSTM d-vector

ECAPA-TDNN 之类更强，但 GE2E 风格的 LSTM d-vector 结构简单、依赖少、在 CPU 上也能
快速跑通，适合作为早期基础。embedding 做 L2 归一化落在单位球面上，便于用余弦相似度
衡量音色接近程度，也方便后续接入对比 / GE2E 损失。接口保持稳定，之后替换成更强的
编码器不影响下游。

## 条件注入方式

文本条件 μ（逐帧）与说话人 embedding（沿时间广播）直接在通道维和 x_t 拼接，
而不是用 FiLM / cross-attention。拼接实现最简单，在小模型上也够用；如果之后发现
说话人可控性不足，再考虑更强的注入方式。

## 已知的坑与未完成项

- **时长对齐**：目前没有实现 MAS 等对齐算法，`Trainer` 只训练声学模型本身，
  逐帧条件需要外部提供。端到端训练是下一步重点。<!-- TODO: 接入单调对齐搜索 -->
- **英文 G2P**：自带实现很粗糙，只为打通链路；追求发音质量应接入 CMUdict / g2p_en。
- **声码器**：默认走 Griffin-Lim，音质有限；HiFiGANLite 需要单独训练。

## 采样步数与质量

流匹配的欧拉步数 `n_timesteps` 是质量 / 速度的主要旋钮。经验上个位数步数即可，
`benchmarks/bench_sampling_steps.py` 用来量化步数对实时率（RTF）的影响。
